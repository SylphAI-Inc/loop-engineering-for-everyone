What Makes Loops Possible

  • Goal-driven work A verifiable objective the loop can continuously measure itself against.
  • Self-validation Browser use and computer use allow agents to close their own testing loops—something previously impossible.
  • Multi-agent roles Separating Evaluator from Builder to prevent "context rot"—leveraging different models, isolated context windows, and specialized prompts.

07 / 15

Loops of Today

  • /goal, /loop (Claude Code) or /cron (AdaL) [repeated or scheduled work — local intervals and cloud routines]
  • permission bypass — --dangerously-skip-permissions (Claude) or --yolo (AdaL)
  • dynamic workflows — agents that adapt as they run
  • Parallel Multi-agent — typically with subagents driven by a main agent
  • Agent Teams (Claude) — full agents focused purely on parallel work execution

The user still designs the system: how many workers to spawn, how they communicate, when they hand work off, how they evaluate one another, how they recover from failures, and when to stop the loop.

08 / 15

What Makes Loops Inaccessible

High complexity Reliable loops require coordinating goals, tools, workers, validation, recovery, state, and stop conditions.
High cost More capable models, longer contexts, browser use, repeated validation, and multiple workers compound runtime cost.
